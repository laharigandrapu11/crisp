# Stage 2 — Adaptive Huffman Compression Microservice

A **from-scratch implementation of Vitter's Algorithm V** (adaptive Huffman coding), exposed as a FastAPI microservice with lossless decompression. Part of the CRISP 2-stage neural compression pipeline.

No `zlib`, `gzip`, `bz2`, `lzma`, or any other pre-built compression library is used. Every bit of the encoder, decoder, tree maintenance, and bit-packing is hand-rolled.

---

## Overview

This service takes UTF-8 text (typically the output of the Stage 1 OCR service) and emits a compressed byte payload that can be perfectly recovered. It uses **adaptive Huffman coding** rather than static two-pass Huffman, which means:

- **Single-pass.** The encoder builds the code table as it reads input — no preliminary frequency scan required.
- **No dictionary overhead.** The decoder reconstructs the same tree from the same byte stream, so nothing needs to be transmitted alongside the payload.
- **Byte-level alphabet.** Input text is UTF-8 encoded and compressed at the byte level (256 + 1 symbols, including the NYT escape), so any Unicode input works without special handling.

The specific variant is **Vitter's Algorithm V** (J. S. Vitter, *"Design and Analysis of Dynamic Huffman Codes"*, JACM 1987), which maintains the sibling property with implicit block numbering and is provably within one bit per symbol of the information-theoretic optimum.

## API Reference

### `GET /health`

Liveness probe.

```bash
curl http://localhost:8001/health
```

```json
{ "status": "ok", "algorithm": "vitter-algorithm-v" }
```

### `POST /compress`

Compress a UTF-8 string. Returns the compressed payload (base64) plus six metrics and the final tree state for visualization.

```bash
curl -X POST http://localhost:8001/compress \
     -H "Content-Type: application/json" \
     -d '{"text": "abracadabra"}'
```

Response (truncated):

```json
{
  "payload_base64": "7CYhyAxiZIQA",
  "metrics": {
    "original_bytes": 11,
    "compressed_bytes": 9,
    "compression_ratio": 1.222222,
    "entropy": 2.040373,
    "avg_bits_per_symbol": 5.636364,
    "encoding_efficiency": 0.362002
  },
  "code_map": { "a": "1", "b": "000", "r": "001", "c": "011", "d": "0101", "NYT": "0100" },
  "tree_structure": { "name": "root", "weight": 11, "number": 513, "children": [ ... ] }
}
```

> **Note.** `code_map` reflects the **final** tree state. Because the tree adapts after every byte, codes used mid-stream differ from the final mapping — this is expected and is exactly what makes the algorithm single-pass.

### `POST /decompress`

Inverse of `/compress`. Takes the `payload_base64` from a compression response, returns the original text.

```bash
curl -X POST http://localhost:8001/decompress \
     -H "Content-Type: application/json" \
     -d '{"payload_base64": "7CYhyAxiZIQA"}'
```

```json
{ "text": "abracadabra" }
```

### `POST /compress/steps`

Returns a step-by-step trace of the encoder — for each input byte: the full tree state, current code map, which swaps Vitter performed, and whether the byte was new. Used by the frontend visualizer and for debugging.

```bash
curl -X POST http://localhost:8001/compress/steps \
     -H "Content-Type: application/json" \
     -d '{"text": "aab"}'
```

### End-to-end round-trip in one shell snippet

```bash
TEXT='hello, world'
PAYLOAD=$(curl -s -X POST http://localhost:8001/compress \
  -H "Content-Type: application/json" \
  -d "{\"text\":\"$TEXT\"}" | python -c "import sys,json; print(json.load(sys.stdin)['payload_base64'])")

curl -s -X POST http://localhost:8001/decompress \
  -H "Content-Type: application/json" \
  -d "{\"payload_base64\":\"$PAYLOAD\"}"
# → {"text":"hello, world"}
```

## Wire Format

The payload returned by `/compress` is a base64-encoded byte string with the following layout:

```
┌─────────────┬─────────────────────────────┬──────────────┐
│ 3 bits      │ N data bits (Vitter stream) │ 0..7 padding │
│ pad_length  │                             │   zero bits  │
└─────────────┴─────────────────────────────┴──────────────┘
```

- The first **3 bits** of the payload are `pad_length`, the number of trailing zero bits appended to reach a byte boundary (0–7).
- The next **N bits** are the Vitter-encoded stream. For known symbols this is the current path from root; for unseen bytes, it's the path to NYT followed by the raw 8-bit byte value.
- The final **`pad_length` bits** are zero padding and are discarded during decoding.

This format lets the decoder recover the exact bit count without storing it as a separate length field. See `huffman/bitpack.py` for the ~15-line implementation.

## Metrics

Every `/compress` response includes six metrics (computed in `huffman/metrics.py`):

| Metric | Definition |
|---|---|
| `original_bytes` | Byte length of the UTF-8-encoded input. |
| `compressed_bytes` | Byte length of the packed output (including the 3-bit header). |
| `compression_ratio` | `original_bytes / compressed_bytes`. Values >1 mean compression. |
| `entropy` | Shannon entropy H(X) = −Σ p(x)·log₂ p(x) of the input byte distribution. Theoretical lower bound on bits/symbol. |
| `avg_bits_per_symbol` | `total_bits_emitted / num_symbols`. What the encoder actually used. |
| `encoding_efficiency` | `entropy / avg_bits_per_symbol`, clamped to [0, 1]. How close the encoder got to the Shannon bound. |

## Testing

```bash
cd stage2_huffman
pytest
```

The test suite covers:

- **Tree mechanics** (`test_tree.py`) — split-NYT creates correct structure, swaps preserve parent pointers, block-leader search returns highest-numbered node, Vitter invariants hold after every operation.
- **Encode/decode round-trips** (`test_vitter.py`) — hand-traced classics (`abracadabra`, `mississippi`), every byte value 0–255 once and repeated, random ASCII up to 10,000 chars, random binary, edge cases (single byte, all-same, nulls, high bytes).
- **Bit-packing** (`test_bitpack.py`) — pad-length header correctness for data lengths 0 through ~500, exact byte-boundary handling, empty input.
- **Metrics** (`test_metrics.py`) — entropy of a constant string is 0, entropy over uniform 256 bytes is 8, ratio/efficiency boundary cases.
- **Service** (`test_service.py`) — endpoint contracts, base64 validity, metrics shape, error paths for malformed payloads.

Invariants are re-validated after every single encode step in the step-by-step test, which catches any tree corruption immediately.

## Design Notes

**Why Vitter Algorithm V over static two-pass Huffman?**
A two-pass encoder must transmit the code table alongside the payload, which adds overhead and requires the sender to know the full input ahead of time. Vitter V is streamable and self-synchronizing — the decoder rebuilds the same tree from the same byte stream, so nothing extra ever needs to be sent.

**Why byte-level rather than character-level?**
UTF-8 text may contain multi-byte characters (e.g. `"résumé"`, `"日本語"`). A byte-level alphabet handles these transparently without any Unicode-specific logic: every possible input byte is a valid symbol, and the output is always a valid UTF-8 string after decode.

**What is NYT?**
Not-Yet-Transmitted — a single special leaf representing "a byte value we haven't seen before". When the encoder hits a new byte, it emits the path to NYT followed by the raw 8-bit byte, then splits NYT into an internal node with the new leaf and a fresh NYT. This is what makes the algorithm single-pass without a preamble.

**Why MAX_NODES = 513?**
A byte alphabet has 256 possible symbols. With the persistent NYT leaf, the tree can hold at most 257 leaves and 256 internal nodes (257 − 1), totalling 513 nodes. Using 511 (the static Huffman number) would cause node numbers to go negative when all 256 byte values appear in a single input.

## Performance Characteristics

Adaptive Huffman has well-known properties that are useful to know when reading the metrics:

- **Short inputs compress poorly.** The tree needs several bytes to adapt; a 4-character string may yield ratio <1 because the NYT overhead dominates.
- **Repetitive inputs compress very well.** `"aaaaaa" * 200` produces a ratio above 3× in our tests.
- **High-entropy (random) inputs barely compress.** A 2,000-byte random sequence stays within 0.7–1.2× — this is the Shannon limit, not an implementation flaw.
- **Complexity.** Encode and decode are both **O(n log k)** where *n* is the input length and *k* is the alphabet size (≤257). In practice, throughput is CPU-bound and adequate for interactive use; see the parent `../README.md` for end-to-end latency numbers.
